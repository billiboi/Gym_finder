# Project Working Notes

## Role and quality bar

- Act as a senior full stack developer with 20 years of experience.
- Treat UI and UX quality as a primary product requirement, not a cosmetic pass.
- Every frontend change must work well on desktop and mobile.
- Prefer practical, focused improvements over broad rewrites.

## Product direction

Palestre in Zona should feel fast, clear, trustworthy, and easy to use for people comparing gyms, disciplines, locations, contact details, opening hours, and maps.

The home page must prioritize immediate search and discovery. Users should understand the product and start searching within the first screen, especially on mobile.

## Performance and production fixes (2026-07-15)

Separate initiative from the UI/UX priorities above, triggered by a Vercel runtime log review: every page showed `cache=MISS` (including robots.txt/sitemap.xml despite already having a `Cache-Control` header), `/admin/richieste` and `/admin/qualita` were 500ing, and favicon/touch-icon requests were 404ing.

1. Directory page caching. Every route was pure dynamic SSR — no `prerender`/`config` export anywhere in `src/routes`. Added `export const config = { isr: { expiration: N } }` (adapter-vercel 6.3.3) to the 11 quasi-static directory routes: 1h (3600s) for `/`, `/zone`(+`[slug]`), `/discipline`(+`[slug]`, +`[slug]/[citySlug]`), `/palestre/[slug]` — anything tied to gym contact/hours freshness; 6h (21600s) for `/chi-siamo`, `/guide`(+`[slug]`), `/per-le-palestre` — editorial content. Deliberately left dynamic/uncached, contrary to the original brief's assumption: `/rivendica-scheda` (its `load()` reads transient query-string state from an email-verification redirect — caching by URL would leak one visitor's confirmation state to another) and `/dashboard-proprietario/[token]` (a per-owner dashboard with form actions — added to the "never cache" list alongside `/admin/*`). `robots.txt`/`sitemap.xml` got `s-maxage`/`stale-while-revalidate` added alongside their pre-existing `max-age` (which alone isn't respected by shared/CDN caches). Status: done, confirmed working directly against `www.palestreinzona.it` — MISS on first hit, HIT by the second/third hit, on every target route; `/rivendica-scheda` correctly stays MISS every time.
2. `claim_requests.updated_at` missing column, causing the 500s above. Root cause: the known Supabase SQL-editor copy-paste bug with `timestamp with time zone` (see `supabase/migrations/README.md`) silently dropping part of an `alter table` statement on a manual production apply — this is the second time it's happened on this same table (resolved gaps dated 2026-07-11 and 2026-07-15 in that README). Fixed with `20260715_001_claim_requests_updated_at_repair.sql`, an idempotent repair migration using `timestamptz` throughout to avoid the same paste bug a third time. Applied to staging then production on 2026-07-15, verified both times by replicating the app's exact `readClaimRequestsListFromSupabase` query against each project (200 OK both times); `public.gyms` row count confirmed unchanged on production (692 before and after). Status: done.
3. Favicon/touch-icon 404s. `static/` only ever had `favicon.svg` — no `.ico`, `.png`, or apple-touch-icon, so browser/iOS auto-probes for those paths 404ed through the SvelteKit catchall function. Found in the process: `favicon.svg` itself was off-brand, still the old green gradient from before the navy/teal/lime rewrite (`36995f4`) — corrected to the current solid-navy pin+dumbbell mark (same geometry as `Logo.svelte`). `<link>` tags for `favicon.ico`, `favicon.png`, and `apple-touch-icon.png` added to `app.html`. Status: blocked on the raster export — this environment has no SVG rasterization tooling (no ImageMagick, no Python/PIL, nothing like `sharp`/`resvg` in `package.json`), and adding a dependency just for a one-off export wasn't approved. Waiting on the user to export `favicon.ico` (16/32/48 multi-size), `favicon.png`, and `apple-touch-icon.png` (180×180) from the corrected `favicon.svg` and drop them into `static/` — the `<link>` tags already reference them, no further code change needed once they land.

## Implementation preferences

- Preserve existing SvelteKit and Tailwind-style patterns unless there is a clear reason to change them.
- Keep edits scoped to the requested priority.
- UI work must never touch production data, Supabase tables, dataset files, import/export scripts, sync scripts, or admin data flows unless the user explicitly asks for a data change in that same request.
- Treat the manually reviewed Supabase gym catalog as production data and as the source of truth. Never overwrite it from local JSON/CSV files as part of UI, UX, SEO, accessibility, or performance work.
- Before any data-changing operation against Supabase or production-like data, create a local export first, verify the export record count, state the before/after counts, and get explicit confirmation for the exact destructive action.
- Never run a `DELETE` + bulk `INSERT` replacement on production data as a convenience path. If a full replacement is truly required, it needs a named local backup, a reviewed source file, a dry run, explicit confirmation, and a post-restore verification.
- Keep local backups of production data out of git, but do create them before touching data. Backup filenames must include the data source and timestamp.
- If the task is UI-only and a data issue appears, stop and report it instead of trying to "fix" data opportunistically.
- Avoid landing-page fluff. Build usable product UI first.
- Avoid redundant UI sections. If a control appears in the hero, later sections should add new capability or context rather than repeat the same task.
- Apply Stop Slop criteria to all UI copy and explanations: cut filler, avoid formulaic three-card sections, use direct active language, remove vague claims, and keep only text that helps the user act.
- Use stable responsive constraints so mobile layouts do not overflow or shift unexpectedly.
- Run `bun run check` after frontend changes.
- If possible, verify locally on desktop and mobile viewport sizes.
